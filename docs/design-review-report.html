5核心流程

5.1 内容管理流程

1
创建内容

在 wiki/public/ 或 wiki/private/ 下创建 .qmd 文件，编写 Frontmatter 和内容

2
设置可见性

通过 Frontmatter 的 visibility 字段标记 public/private

3
本地预览

运行 quarto render 本地预览完整内容（含私人）

4
推送到 GitHub

git push 触发 GitHub Actions

5
CI 自动过滤

filter-wiki.py 扫描并过滤 private 内容

6
发布到 GitHub Pages

仅公开内容渲染并发布到 gh-pages

8风险看板

P0 私人内容误推送

CI 过滤脚本 BUG 或配置错误，导致 private 内容推送到 GitHub Pages
影响: 隐私泄露，不可逆
缓解: ① filter-wiki.py 需编写单元测试 ② CI 完成后进行端到端验证 ③ 添加 pre-push hook 本地检查
工作量: 0.5天(测试) + 0.5天(验证)

P1 过滤脚本维护困难

随着 wiki 内容增长，过滤逻辑可能变得复杂
影响: 可维护性下降，未来迭代成本增加
缓解: ① 保持脚本简单，遵循单一职责 ② 添加详细注释 ③ 使用 type hints
工作量: 0.5天(重构)

P1 Wiki 内容膨胀

缺乏整理机制，内容越来越杂乱，难以维护
影响: 知识库价值降低，变成"知识垃圾桶"
缓解: ① 定期 lint (参照 Karpathy 方案) ② 使用 tags 分类 ③ 设置归档策略
工作量: 0.5天(lint 脚本) + 持续维护

P2 LLM 生成质量不可控

LLM 生成的 wiki 内容可能存在错误或偏见
影响: 知识准确性下降
缓解: ① 关键内容人工审核 ② 设置 accuracy check 流程
工作量: 持续

P2 搜索功能缺失

当前方案无搜索，随着内容增长难以找到需要内容
影响: 使用效率降低
缓解: ① 使用 Obsidian 客户端搜索 ② 后期接入 qmd 或其他搜索方案
工作量: 1天(可选)
